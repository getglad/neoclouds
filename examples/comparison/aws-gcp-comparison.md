# AWS vs GCP — Security Posture Comparison

**Evaluated**: 2026-06-01
**CSV outputs**:
- [`comparison-matrix.csv`](./comparison-matrix.csv) (side-by-side, 21 capability dimensions)
- [`aws-deep-dive.csv`](../aws/aws-deep-dive.csv) (122 probes) · [`gcp-deep-dive.csv`](../gcp/gcp-deep-dive.csv) (115 probes)
**Per-platform reports**: [`aws-assessment.md`](../aws/aws-assessment.md) · [`gcp-assessment.md`](../gcp/gcp-assessment.md)

**Context (identical for both):** a multi-team organization protecting its **own proprietary IP** (source, model weights, internal docs), running AI/GPU + general compute + serverless across many accounts (AWS) / projects (GCP) under one organization. Both evaluated as absolute deep-dives against well-architected best practice — *not* against each other for a "winner," but to locate where each sits and where they diverge.

## Top-line summary

**The two platforms cluster together far more than they diverge, and they cluster on the same thesis: the capability is present, the secure configuration is opt-in.** The status distributions are near-identical — AWS **70 present / 46 partial / 3 misleading / 3 absent** (122 probes), GCP **64 present / 44 partial / 2 misleading / 5 absent** (115 probes). Roughly 57% of probes on each platform are "present," ~38% "partial," and each carries a large "Documented but Misleading" tail (AWS 24, GCP 34) — the familiar-term-weaker-default pattern. **Neither platform is missing controls; both ship permissive defaults and leave the secure posture to the customer.** The single most important finding is the *same hole on both*: read-access to the crown jewels is invisible by default (AWS S3 object-level **data events** off; GCP **Data Access** logs off). If you remember one thing across both clouds, it is "turn on read-access logging on your IP stores before go-live."

**Where they genuinely diverge is structure, not capability.** GCP's **Org→Folder→Project** hierarchy gives lightweight, inheritable isolation units that fit a multi-team org more naturally than AWS's heavier account-per-boundary model. GCP ships its data-exfiltration perimeter as **one coherent control (VPC Service Controls)**, whereas AWS makes you **compose a triad** (SCP + RCP + VPC-endpoint policy) — with RCPs (Nov 2024) only recently closing the resource-policy side. GCP has a **native deploy-time image-signing gate (Binary Authorization)**; AWS has signing (Signer) but **no native gate** — you assemble admission control yourself. And GCP's **Access Transparency / Access Approval** (logging and gating Google's *own* staff access to your content) has no default AWS equivalent. AWS's edges are the **breadth and maturity of the composable IAM/STS primitives**, the **Nitro** hardware-isolation story, and an organizational/compliance machine that is second to none. Neither is "more secure"; they are differently shaped, and a team's existing fluency in one model is a legitimate tiebreaker.

**What surprised the evaluator:** both fact-check passes overturned a similar number of confident-but-stale claims (AWS 10, GCP 13), dominated by *defaults that changed* and *people/dates* — AWS's was a **reversal** (CodeCommit returned to GA Nov 2025), GCP's clustered on the **secure-by-default baseline date/enumeration** and the **CISO seat change**. And the matrix's one "both-misleading" cell — **image signing** — is the most instructive: GCP has the better underlying capability (a native gate) yet lands at the same honest verdict as AWS because that gate ships **opt-in/dry-run**. Capability and default are different axes, and the default is where incidents live.

> Read the matrix as the scannable layer; the nuance lives in the prose and the per-platform deep-dives. A cell can read "present" on both while the *defaults* differ, or "misleading" on both while the *capabilities* differ (image signing is exactly that).

## Where each platform sits

**AWS** — Best signal: **Identity Granularity** — action-level IAM with the richest set of composable scoping primitives (STS, session policies, session tags, source identity, permission boundaries, ABAC) and a mature workload-identity/federation story. Best non-obvious strength: **Transparency** — AWS explicitly documents its non-guarantees (Shared Responsibility Model; "API Gateway API keys are not authorization"; "root cannot be constrained by IAM"). Worst signal: **default-on visibility** — CloudTrail's default is a 90-day management-event buffer; S3 object reads and most threat-detection plans (GuardDuty) are opt-in/paid. The one thing a practitioner must know: **the data perimeter is a triad you assemble** (SCP + RCP + VPC-endpoint policy), and account sprawl is the cost of the isolation model.

**GCP** — Best signal: **Organizational Substrate** — the Org→Folder→Project hierarchy plus a rich Org Policy Service is the model others copy, and it fits multi-team isolation cleanly; a **secure-by-default baseline** even auto-applies to new orgs (since May 3, 2024). Best non-obvious strength: **provider-access transparency** — Access Transparency/Access Approval. Worst signal: same as AWS — **default-on visibility** (Data Access logs off) — plus two signature credential footguns (**basic roles** and **service-account JSON keys**). The one thing a practitioner must know: **the secure-by-default baseline is not retroactive** — an organization created before May 2024 inherits SA-key-allowed and default-SA-Editor, and must enforce the baseline manually.

## Recommendations

The per-platform reports carry the full P0/P1/P2 lists with account-runnable audit checks ([`aws-audit-checklist.csv`](../aws/aws-audit-checklist.csv), [`gcp-audit-checklist.csv`](../gcp/gcp-audit-checklist.csv)). The **cross-platform throughline** — the four moves that matter on *both* clouds, with different mechanisms — is:

1. **Turn on read-access logging on IP stores before go-live.** AWS: S3 object-level data events on a durable, immutable org trail. GCP: Data Access (DATA_READ) audit logs + an immutable aggregated org sink. *This is the #1 gap on both.*
2. **Build and enforce the exfiltration perimeter.** AWS: compose SCP (identity) + RCP (resource, `aws:PrincipalOrgID`) + VPC-endpoint policies. GCP: enforce (not dry-run) VPC Service Controls covering every IP-bearing service.
3. **Eliminate standing credentials and over-broad roles.** AWS: retire IAM access keys, enforce IMDSv2, centralize root. GCP: disable SA-key creation + adopt WIF, stamp out basic roles and the default-SA Editor grant (manually on pre-May-2024 orgs).
4. **Turn on the paid/opt-in detection tier.** AWS: GuardDuty protection plans + Security Hub org-wide. GCP: Security Command Center Premium/Enterprise. The free baseline on both is insufficient for IP-theft detection.

**Platform-specific must-dos:** on **GCP**, if your organization predates **May 3, 2024**, treat the secure-by-default baseline as *not applied* and roll it out across the hierarchy; turn **Binary Authorization** to enforce-mode (you have the native gate — use it); enable **Access Transparency/Access Approval**. On **AWS**, assemble the **RCP** leg of the data perimeter (verify service coverage) and build the **image-verification admission control** GCP gives you natively; convert the legacy **IMDSv1** fleet and retire **long-lived access keys**.

**Tiebreaker guidance (not a verdict):** for a greenfield multi-team org that values *lightweight isolation, a unified exfil perimeter, native signing/provenance, and provider-access transparency*, GCP's structure is a natural fit. For an org that values *the deepest IAM/STS primitive surface, the broadest compliance/IR machine, and Nitro-grade hardware isolation*, AWS is. Both require the same four hardening moves above before proprietary IP lands — the platform you already operate fluently is a legitimate deciding factor, because **the dominant risk on both is misconfiguration of an opt-in control, not a missing one.**
